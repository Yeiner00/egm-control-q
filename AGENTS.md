# Repository Guidelines

## Project at a Glance
EGM Admin: Vite + React 18 + TypeScript SPA for zarpes (maritime departures), vehicle/boat report management, statistics, and an Excel-report importer (V2). UI is Tailwind + shadcn/ui primitives, auth and data via Supabase, serverless routes on Vercel, and one Deno Edge Function (`resolve-import-names`) for Gemini-powered name resolution.

## Project Structure & Module Organization
- `src/pages`: top-level routes — `Index.tsx`, `Login.tsx`, `NotFound.tsx`.
- `src/components`: feature components — `ZarpeForm.tsx`, `ZarpesTab.tsx`, `ReportesTab.tsx`, `EstadisticasTab.tsx`, `EstadisticaTab.tsx`, `InicioTab.tsx`, `HeaderMiniCalendar.tsx`, `ThemeToggle.tsx`, `ImageUploader.tsx`, etc.
- `src/components/estadisticas/`: report-management flow — `ReportUploader`, `BoatReportForm`, `VehicleReportForm`, `ReportListRow`, `ReportImportV2`, `ReportCatalogAdmin`, dashboards, exports.
- `src/components/ui/`: shadcn/ui primitives; regenerate rather than hand-edit unless you are intentionally changing them.
- `src/lib/`: pure helpers grouped by domain (`report*`, `statistic*`, `officers`, `motives`, `sites`, `aiRateLimit`, `xlsx`, etc.). Most files ship a colocated `*.test.ts`.
- `src/hooks/`: app-level React hooks (currently only `useUserInvalidAria.ts`).
- `src/integrations/supabase/`: `client.ts` is autogenerated (do not hand-edit); regenerate `types.ts` via the Supabase CLI when the DB schema changes.
- `src/test/`: shared Vitest setup (`setup.ts`) plus a few cross-cutting tests.
- `api/`: Vercel serverless handlers, each exporting `default { fetch }`. The same files are also served by a Vite dev middleware — see "Local dev gotchas".
- `supabase/functions/`: Deno Edge Functions. `resolve-import-names` is the only one the app calls; `extract-report`, `extract-zarpe`, `generate-statistic-cells`, `summarize-novedades` are adjacent tooling.
- `supabase/migrations/`: append-only, timestamp-prefixed SQL. `motive_normalization_diagnostics.sql` and `report_number_diagnostics.sql` are ad-hoc helpers, not migrations.
- `public/`: static assets; `index.html` is the SPA shell.

## Build, Test, and Development Commands
- `npm install`: install deps. Lockfile is `package-lock.json` — use npm, not pnpm/yarn.
- `npm run dev`: Vite dev server on `http://localhost:8080` (host `::`, HMR overlay disabled).
- `npm run build`: production build into `dist/`.
- `npm run build:dev`: build with `--mode development` (keeps dev-only Vite behavior; useful for staging-like builds).
- `npm run preview`: serve the built app locally.
- `npm run lint`: ESLint flat config (`eslint.config.js`).
- `npm run test`: Vitest once (`vitest run`).
- `npm run test:watch`: Vitest in watch mode.
- `npx playwright test`: Playwright is configured via `lovable-agent-playwright-config` (see `playwright.config.ts` and `playwright-fixture.ts`) but no npm script is wired — invoke the binary directly.
- No standalone `tsc` script. Type-check through `npm run build` (uses `tsc -b` via the Vite pipeline) or your editor's TS server. `tsconfig.app.json` is intentionally lenient (`strict: false`, `noUnusedLocals: false`, `noImplicitAny: false`); do not tighten it without being asked.

## Local Dev Gotchas (read before touching `api/` or the import flow)
- `vite.config.ts` registers a `reportImportDevApi` middleware that SSRs `api/report-imports.ts` and `api/report-imports/[id]/confirm.ts` against `/api/report-imports` and `/api/report-imports/:id/confirm`. **HMR does not reload server middleware — restart `npm run dev` after editing these files.**
- The same files deploy to Vercel as serverless functions. In dev, env vars are loaded from `.env`; in production Vercel injects them. Any change must work in both modes.
- The Excel V2 importer (`src/components/estadisticas/ReportImportV2.tsx` + the two `api/report-imports*` files + the Deno function) depends on Supabase tables `officers`, `officer_aliases`, `report_motive_catalog`, `report_motive_aliases`, `report_site_catalog`, `report_site_aliases`, `report_import_jobs`, `report_import_fields`, `report_import_alias_suggestions`, `report_import_person_suggestions`, `report_import_catalog_suggestions`, and a storage bucket named `report-import-snapshots`. Missing the bucket surfaces as a clear error from the upload handler, but missing tables will not — verify migrations are applied.
- The Vercel function enforces a per-user rate limit of 8 uploads/min (`RATE_LIMIT_MAX_UPLOADS` in `api/report-imports.ts`); a 429 means back off.

## Environment & Secrets
Copy `.env.example` to `.env.local` (it is the only `.env*` not blocked by `.gitignore`). Variables in use:

- `VITE_SUPABASE_URL`, `VITE_SUPABASE_PUBLISHABLE_KEY`: required by the Supabase client at runtime.
- `VITE_AI_MIN_REQUEST_INTERVAL_MS`, `VITE_AI_DAILY_REQUEST_LIMIT`, `VITE_AI_MINUTE_REQUEST_LIMIT`: client-side AI rate-limit thresholds consumed by `src/lib/aiRateLimit.ts`.
- `SUPABASE_URL`, `SUPABASE_PUBLISHABLE_KEY`: required by the Vercel functions in `api/`.
- `GEMINI_API_KEY`, `GEMINI_TEXT_MODEL` (default `gemini-3.1-flash-lite`), `GEMINI_VISION_MODEL`: used only by the Deno Edge Function `supabase/functions/resolve-import-names`. The free-tier limit cited in the code is 15 RPM / 250K TPM / 500 RPD; the function returns 429 with `retryAfterMs` on rate-limit and 402 on credits-exhausted. The function also sanitizes any returned `officerCedula` against the candidates list per field — do not loosen that constraint.
- `CRON_SECRET`: bearer shared with `/api/supabase-keepalive`, which Vercel invokes daily at `0 10 * * *` (see `vercel.json`). The handler calls the `keepalive` RPC (added by migration `20260514103000_add_keepalive_rpc.sql`).

Never commit real values. `.gitignore` already excludes `.env`, `.env.*` (with `!.env.example`).

## Coding Style & Naming Conventions
- TypeScript + functional React components with hooks.
- 2-space indent, semicolons, double quotes, single-statement arrow functions where they help.
- `PascalCase` for components, `camelCase` for utilities, route files in `src/pages`.
- Prefer the `@/` import alias (`@/lib/...`, `@/components/...`, `@/integrations/...`).
- Tailwind utility classes plus the design tokens in `src/index.css` (colors use `hsl(var(--token))` defined in `tailwind.config.ts`); add new colors as CSS variables, not hard-coded values.
- Use `cn()` from `src/lib/utils.ts` (clsx + tailwind-merge) for conditional class composition — not template strings.
- The UI is in Spanish by design; do not "translate" strings to English.

## Testing Guidelines
- Vitest + Testing Library + `jest-dom` (`vitest.config.ts`, `src/test/setup.ts`).
- Tests live next to the code: `src/lib/foo.ts` → `src/lib/foo.test.ts`. Cross-cutting tests go in `src/test/`.
- Global Vitest types are enabled via `tsconfig.app.json` (`types: ["vitest/globals"]`), so `describe`, `it`, `expect` are imported implicitly in tests.
- No coverage threshold; focus on helpers, form/data transforms, and zod schema validation (zod schemas live with the modules they describe, e.g. `reportImportSchema.ts`).
- For component tests, `setup.ts` already stubs `matchMedia`, `ResizeObserver`, and `Element.prototype.scrollIntoView`.

## Supabase & Backend Conventions
- Project id (from `supabase/config.toml`): `vsiyibdyuwgsejcwjeht`. Linked to a Vercel project named `egm-admin` (see `.vercel/project.json`).
- Migrations are append-only; never edit a migration that has been applied. Add a new file with a later timestamp prefix.
- RLS is enabled across the schema; new tables must include policies (see `20260514054000_harden_supabase_schema.sql` and `20260514054500_restrict_rls_auto_enable.sql`).
- The Deno Edge Function `resolve-import-names` enforces that returned `officerCedula` values belong to the candidates list per field.

## Deploy & Cron
- `vercel.json` defines one cron (`/api/supabase-keepalive` at `0 10 * * *`) and the SPA rewrite (everything not under `/api/*` → `/index.html`).
- `api/supabase-keepalive.ts` requires `Authorization: Bearer ${CRON_SECRET}`; Vercel injects the header automatically for cron invocations.

## Commit & Pull Request Guidelines
- Branch: `main`. Remotes: `origin` and `original` (the latter is the upstream mirror).
- Short, imperative commit messages scoped to one change (e.g. `Add zarpe export validation`).
- PR description should call out: impacted screens/flows, linked issue, screenshots for UI changes, and any Supabase migration or Edge Function change.
- Do not commit secrets or `.vercel/` contents. Stale dev logs (`vite-dev*.log`, `.codex-vite*.log`, `debug.log`) may be present in the working tree as untracked files — leave them alone unless cleaning up.
